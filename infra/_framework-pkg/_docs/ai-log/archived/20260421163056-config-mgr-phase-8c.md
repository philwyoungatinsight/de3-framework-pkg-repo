# config-mgr Phase 8C — de3-runner framework port

## Decision

User confirmed: every deployment repo depends on its own default-pkg, so de3-runner's
`infra/default-pkg/` is used when de3-runner is run standalone. Option A (full port)
is correct.

## What changed (de3-runner repo, commit 5f4479f)

All changes are a direct copy from pwy-home-lab-pkg's updated framework. No logic
changes — the port is mechanical.

- `_config-mgr/` tool added (full Python package + generate/run entry points)
- `framework_config_mgr.yaml` added (merge_method: interleave, output_mode: normal)
- `set_env.sh`: exports `$_CONFIG_DIR`, calls `config-mgr generate` on source
- `root.hcl`: reads from `$_CONFIG_DIR` (plain YAML), removes `sops_decrypt_file()`
- `validate-config.py`: RULE 6 — config_source chain checks (no-op on de3-runner since
  no packages declare config_source, but forward-compatible)
- `generate_ansible_inventory.py`: reads from `$_CONFIG_DIR` glob
- `config_base` role: plain `include_vars` from `$_CONFIG_DIR`, no SOPS
- `unit-mgr main.py`: `_read_framework_packages()` + `_resolve_config_source()` added;
  Phase 3 uses config_source resolution when routing config file reads/writes

## Behaviour for de3-runner standalone

de3-runner's `framework_packages.yaml` has no `config_source` entries. config-mgr
generate is a pass-through: copies each package's own config to `$_CONFIG_DIR` without
merging. The framework is forward-compatible — importing repos can declare config_source
and it will be resolved correctly.
