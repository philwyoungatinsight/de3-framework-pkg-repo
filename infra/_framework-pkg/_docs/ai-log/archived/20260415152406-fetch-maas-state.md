# fetch-maas-state: unified MaaS state snapshot

## What was built

Two new files to reduce redundant MaaS API queries and give humans/code a single place to reason about machine state:

### `scripts/ai-only-scripts/fetch-maas-state/run`

Standalone script that:
1. SSHes to the MaaS server (host read from `maas-pkg.yaml`)
2. Runs an embedded Python probe script **on the MaaS server** (in parallel per machine) that:
   - `maas maas-admin machines read` — all machine state, interfaces/MACs, IPs
   - `maas maas-admin machine power-parameters <sysid>` — power config, credentials scrubbed
   - TCP socket probes: AMT port 16993, webhook URL, SSH port 22
   - Optionally `maas maas-admin machine query-power-state` (`--probe-power` flag, slow)
3. Converts JSON result to YAML, writes to `$_DYNAMIC_DIR/maas-state/maas-state.yaml`
4. Prints summary table to stdout

Flags:
- `--force` — bypass cache TTL, always re-fetch
- `--probe-power` — include live `query-power-state` (up to 90s/machine via AMT)
- `--ttl N` — override cache TTL (default: `_MAAS_STATE_CACHE_TTL` or 120s)
- `--out PATH` — write to a specific file

### `utilities/bash/maas-state-cache.sh`

Sourceable bash library for other scripts:
- `ensure_maas_state_fresh [host] [ttl]` — fetch if stale
- `maas_state_file` — echo path to cache file
- `maas_state_get_status <hostname>` — echo `status_name`
- `maas_state_get_field <hostname> <field>` — echo any top-level field
- `maas_state_find_sysid <hostname>` — echo `system_id`
- `maas_state_get_probes <hostname>` — echo probes dict as JSON
- `maas_state_all_json` — full machines dict as JSON (for jq/Python)
- `maas_state_list` — pretty table of all machines
- `maas_state_invalidate` — delete cache file

## Why

The existing MaaS automation makes a large number of redundant API calls:
- Each gate playbook calls `machines read` independently
- Lifecycle scripts poll `machine read` every 15–30s
- Power management tests call `query-power-state` per machine serially

This snapshot replaces all enumeration/decision-time queries with a single
parallel fetch. Live polling loops (`commission-and-wait.sh` etc.) still poll
independently since they need real-time state updates.

## State file structure

```yaml
fetched_at: "..."
maas_host: "10.0.10.11"
cache_ttl_seconds: 120
probe_power_state: false
summary:
  total_machines: N
  by_status: {Deployed: 2, Ready: 1}
machines:
  ms01-01:
    system_id: "..."
    status_name: "Deployed"
    power_type: "amt"
    power_state_cached: "on"
    ip_addresses: [...]
    interfaces:
      enp2s0:
        mac_address: "aa:bb:cc:dd:ee:ff"
        links: [...]
    power_parameters:
      power_address: "10.0.11.x"
      port: 16993
      # credentials omitted
    probes:
      amt_tcp_reachable: true
      amt_tcp_host: "10.0.11.x"
      amt_tcp_port: 16993
      amt_tcp_check_ms: 45
      ssh_tcp_reachable: true
      ssh_ip: "10.0.12.x"
```
