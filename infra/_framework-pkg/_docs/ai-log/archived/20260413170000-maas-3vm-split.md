# MaaS 3-VM Split Architecture

**Date**: 2026-04-13  
**Session**: maas-3vm-split

## Problem

The single combined MaaS VM (`maas-server-1`) was causing repeated API outages during
PXE provisioning. Root cause: the rack controller's RPC stream competed with
DHCP/TFTP/PXE traffic on the provisioning NIC (10.0.12.2). Even with iptables isolation
(previous fix), the problem persisted — suggesting resource contention on the single VM.

The permanent fix is architectural: split into three separate VMs so region, rack, and
database are isolated on dedicated hosts with dedicated NICs.

## Architecture Change

**Before**: Single combined VM `maas-server-1` at 10.0.10.11 (management) + 10.0.12.2
(provisioning), running PostgreSQL + region controller + rack controller.

**After**: Three VMs:

| VM | IP(s) | Role |
|----|-------|------|
| `maas-db-1` | 10.0.10.12/24 | PostgreSQL only |
| `maas-region-1` | 10.0.10.11/24 | MaaS region controller, API (keeps existing IP) |
| `maas-rack-1` | 10.0.10.13/24 + 10.0.12.2/24 | MaaS rack controller, DHCP/PXE |

## Key Design Decisions

- **RPC isolation iptables rules removed**: no longer needed — region and rack are on
  separate VMs with separate NICs; there is no provisioning NIC on the region.
- **IP forwarding + NAT masquerade moves to rack**: the rack is the gateway for VLAN 12,
  so IP forwarding and NAT are configured there (not on region).
- **Region routing for jump box**: configure-rack adds a static route on maas-region-1
  for 10.0.12.0/24 via 10.0.10.13 so it can still serve as an SSH jump box.
- **Rack init uses region secret**: `configure-rack` reads `/var/snap/maas/common/maas/secret`
  from maas-region-1 via `delegate_to` and uses it for `maas init rack`.
- **Old maas-server-1 directory kept**: terragrunt.hcl kept on disk (TF state still
  references it); entry removed from YAML config so it no longer builds/destroys.
  To fully clean up: `tofu state rm` on old units then delete directories.

## State Cleanup Required

The old `maas-server-1` Proxmox VM and `configure-server` null resource have TF state
in GCS. To clean up after the new VMs are verified:

```bash
source set_env.sh
cd infra/pwy-home-lab-pkg/_stack/proxmox/pwy-homelab/pve-nodes/pve-1/vms/utils/maas-server-1
terragrunt state rm proxmox_virtual_environment_vm.this

cd ../../../../../../../../null/pwy-homelab/maas/configure-server
terragrunt state rm null_resource.configure_maas
```

Then delete the directories and the old VM from Proxmox manually.
