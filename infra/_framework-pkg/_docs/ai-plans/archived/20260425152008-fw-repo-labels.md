# Plan: fw-repo-labels — Framework Repo Label System

## Objective

Add a `labels` field to `framework_repo_manager.yaml` repo entries, wiring it through the
scanner data model and text renderer. Labels are a list of YAML objects with a required `name`
field and optional qualifiers (forward-compatible). Reserved labels (underscore-prefixed) carry
a `value` qualifier. Initialize `_purpose` and `_docs` for every repo in
`pwy-home-lab-pkg`'s `framework_repo_manager.yaml`.

## Context

- **Schema-less data model**: the scanner stores repo data as plain Python dicts and emits YAML/JSON.
  Labels slot in alongside `notes`, `url`, `created_by`, etc. — no strict dataclass needed.
- **`notes` pattern**: the codebase already does exactly this for a `notes: [str]` list.
  `_load_repo_manager` reads it, stores it in the `declared_repos` stub, and `_scan_dir`
  back-fills it into the result entry. Labels follow the exact same path.
- **Notes are never rendered**: the existing `notes` field has no renderer support.
  Labels should be rendered in the text output so the `_purpose`/`_docs` values are visible.
- **Framework-pkg version**: currently `1.11.0` in `_provides_capability`; bump to `1.12.0`
  (new feature in fw-repos-visualizer).

### Label schema (design)

```yaml
labels:
  - name: _purpose
    value: "Short description of what this repo does"
  - name: _docs
    value: "https://github.com/philwyoungatinsight/<repo>"
  - name: some-plain-tag   # no extra fields — a tag/marker
```

`name` is the only required field. Any additional keys are optional qualifiers.
Reserved labels (underscore prefix) are expected to carry `value:`. This is forward-compatible:
future qualification (e.g. `owner:`, `tier:`) is just adding more keys.

### `infra/_framework-pkg/_framework/_fw-repos-visualizer/fw_repos_visualizer/renderer.py` — modify

**In `_render_text`, after the repo header line** (currently line 68, `lines.append(f"{repo_name}  ({url})...")`),
add label rendering for any label that has a `value:` key:

```python
lines.append(f"{repo_name}  ({url}){lineage_str}{declared_str}")
# add immediately after:
for lbl in repo_data.get("labels", []):
    lbl_value = lbl.get("value")
    if lbl_value is not None:
        lines.append(f"  [{lbl['name']}] {lbl_value}")
```

This renders reserved labels (which have `value:`) and silently skips plain tag-style labels
(which have no `value:`), keeping the output clean when bare tags are used in the future.

**In `_render_dot`, add `_purpose` to the cluster subtitle** — find the lines building `subtitle`
(around lines 122–127) and append the `_purpose` value if present:

```python
# existing:
subtitle = f"\\n{url}" if url else ""
if show_lineage and created_by:
    subtitle += f"\\ncreated by: {created_by}"
if is_declared:
    subtitle += "\\n[declared only]"

# add after:
for lbl in repo_data.get("labels", []):
    if lbl.get("name") == "_purpose" and lbl.get("value"):
        subtitle += f"\\n{lbl['value']}"
        break
```
