# Plan: maas-snafu-18 — Premature Annihilation + NET-1 Ignores skip_ssh_check (ms01-02)

## What Broke

Three bugs in `maas-lifecycle-gate/playbook.yaml` cause ms01-02 to be repeatedly
annihilated or its deployment to fail even after Rocky 9 deploys successfully and
the login screen appears.

### Bug 1: Grace period missing for Deploying candidates

The annihilation check (PRE only) marks machines stuck in
`Commissioning/Testing/Deploying` with BMC=off as annihilation candidates.
A 60-second grace period + MaaS re-check was added for `Commissioning` candidates
to handle the race where a machine just finished commissioning (curtin powered it off)
and MaaS hasn't yet updated the status to Ready.

The SAME race exists for `Deploying` machines:
- Curtin runs `zz_signal_maas` (MaaS marks Deployed) then `zzz_poweroff`
- For `mgmt_wake_via_plug` machines, AC power is cut after poweroff
- Smart plug reports off → AMT unreachable → BMC=off
- If the PRE gate runs at this moment, it sees: Deploying + BMC=off → candidate

The filter task after the pause ONLY removes machines that transitioned from
`Commissioning` to a non-stuck state. `Deploying` machines are kept in the
annihilation list unconditionally via the `else` branch:
```
{%- else -%}
  {# Not a commissioning candidate — always include #}
  {%- set _ = result.append(item) -%}
{%- endif -%}
```

Even if MaaS has marked the machine `Deployed` by the time the filter runs, the
machine stays in the annihilation list and gets destroyed.

### Bug 2: NET-1 SSH check ignores `skip_ssh_check: true`

ms01-02 has `skip_ssh_check: true` in config because after deploy, the machine
waits at the GRUB screen for an extended period before booting Rocky 9. During
this GRUB wait, SSH port 22 is not open. The `deployed:post` NET-1 check fires,
finds port 22 closed, and fails the gate.

The `skip_ssh_check` field is NOT included in `_all_machines` and NOT filtered in
the NET-1 loop. The flag has no effect.

### Bug 3 (minor): Grace period prompt message doesn't mention Deploying

The `prompt` in the pause task only mentions commissioning candidates by name, so
the log is misleading when Deploying candidates are the trigger.

## Fix

### `playbook.yaml` — Load machine list (Play 1)

Add `skip_ssh_check` to the machine dict in `_all_machines`:
```jinja
'skip_ssh_check': cfg.get('skip_ssh_check', false),
```

### `playbook.yaml` — Grace period tasks (Play 2)

1. Change the "Wait 60s" pause condition from:
   ```
   selectattr('maas_status', 'equalto', 'Commissioning')
   ```
   to:
   ```
   selectattr('maas_status', 'in', ['Commissioning', 'Deploying'])
   ```

2. Change the "Re-read MaaS state" task condition identically.

3. Update the filter task to also handle `Deploying` candidates — check if
   the machine transitioned to `Deployed`:
   ```jinja
   {%- elif item.maas_status == 'Deploying' -%}
     {%- set current_status = recheck.get(item.hostname, item.maas_status) -%}
     {%- if current_status not in ['Deployed', 'Ready'] -%}
       {%- set _ = result.append(item) -%}
     {%- endif -%}
   ```

4. Update pause prompt to list both Commissioning and Deploying candidates.

### `playbook.yaml` — NET-1 loop

Add `rejectattr('skip_ssh_check', 'equalto', true)` to the loop filter:
```jinja
{{ _machines
   | selectattr('cloud_public_ip', 'defined')
   | selectattr('cloud_public_ip', 'ne', '')
   | rejectattr('skip_ssh_check', 'equalto', true)
   | list }}
```

## Verification

After the fix:
- ms01-02 deploys Rocky 9 → powers off → smart plug cuts power → gate runs
- Deploying+BMC=off candidate → 60s wait → MaaS re-check shows Deployed → removed from list
- Deployed:post NET-1: ms01-02 skipped (skip_ssh_check=true) → gate passes
- Build completes without spurious annihilation
